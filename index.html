1. Philosophy

  • green is a library. Users define steps, wiring, and workflows in their own project; green supplies the engine and the CLI plumbing. The entry point is a self-contained babashka script that forwards *command-line-args* to green.
  • Everything is a map. The desired state is a Clojure map loaded from EDN. It threads through every step as opts. Steps read it, act, and return it (possibly enriched). Observed reality (e.g. OpenTofu outputs) merges back into the same map.
  • Idempotent apply. No plan phase. Running the same lifecycle event twice converges to the same result. create and delete are symmetric.
  • Unix exit-code semantics. Steps report success/failure the way processes do: 0 ok, > 0 error. The workflow's final code becomes the process exit code.
  • babashka-compatible, always. Pure Clojure; only dependencies that babashka bundles (Selmer, cheshire, babashka.cli) or resolves as libraries. Everything must run under bb and the JVM alike.

2. Desired state & the opts contract

The desired state is an EDN file — green.edn in the current directory by default, overridable with -f/--file. The CLI reads it into a map, stamps the lifecycle event, and hands it to the workflow as the initial opts.

green reserves the :green/* namespace in opts:

KeyTypeMeaning
:green/eventkeywordThe lifecycle event, stamped by the CLI from the first positional argument (:create, :delete, … — an open set; those two are conventions, not an enum). Steps branch on it.
:green/dry-runbooleanStamped by the CLI's --dry-run flag when present; dry-run advice uses it to skip side-effecting steps.
:green/exitintOutcome of the last step: 0 success, > 0 error. If a step returns a map without it, the runner sets it to 0. The workflow's final :green/exit value becomes the process exit code.
:green/errstringError message. Present when :green/exit > 0.
:green/tracestringStack trace. Present when the failure came from a thrown exception.
:green/branchesvector of mapsAt a join step (or a collapsed failed fork): the resulting opts of every parallel branch. Treat it as a collection; sort by your own key when order matters.

3. Steps

A step is a plain Clojure function from opts to opts, identified by a qualified name (e.g. :zk/node). The qualified name is the anchor for wiring, logging, CLI slices, and advice.

  • Steps are event-aware: the same :infra/tofu step runs tofu apply -auto-approve for any non-:delete event and tofu destroy -auto-approve when :green/event is :delete.
  • Steps signal failure by returning :green/exit > 0, or by simply throwing: the runner wraps every step (including its advice stack) in try/catch and converts a throw into :green/exit 1 (or the :green/exit in exception data) + :green/err + :green/trace. Nothing escapes the contract.
  • A step returning a non-map is an error and is converted the same way.

4. Workflows: wire-fn & next-fn

(require '[green.workflow :as wf])

(def workflow
  (wf/workflow {:start   :zk/start      ;; required: first step
                :end     nil            ;; optional: slice boundary (runs, then stop)
                :wire-fn wire-fn        ;; required: the static graph
                :next-fn next-fn}))     ;; optional: the dynamic router

(wf/run workflow opts)                  ;; => final opts

wire-fn — the static graph

(wire-fn step) returns a vector of the step's function followed by its default successors:

(defn wire-fn [step]
  (case step
    :zk/start   [start-fn  :zk/node]        ;; one successor: sequential
    :zk/node    [node-fn   :zk/zoo-cfg]
    :zk/zoo-cfg [zoo-fn]                    ;; no successor: terminal
    :ci/fork    [fork-fn   :ci/a :ci/b]))   ;; several: run in parallel

The wiring encodes the happy path and the parallelism: multiple successors run concurrently and later converge (see §5).

next-fn — the dynamic router

When present, next-fn is consulted after every step (except the end step) and fully decides what happens next:

(next-fn step default-next opts) ;; => seq of [next-step opts] pairs
  • step — the step that just ran; default-next — the seq of successors from wire-fn (or nil); opts — the step's result, including :green/exit.
  • Return one pair to continue, several to fan out (each pair runs in parallel with its own opts), or nil/empty to terminate the branch.
  • Because it sees :green/exit, next-fn is also the error router: reroute failures to cleanup steps, retry, or halt.

Composition — a workflow is a step

A workflow run has the same opts → opts outcome contract as any step, and wf/step wraps a workflow as an ordinary step function:

(wf/step cluster-wf)                       ;; workflow → step fn: wire it, advise it, fan it out
(wf/step cluster-wf
         {:in  (fn [opts] …)               ;; shape the opts entering the sub-workflow
          :out (fn [opts result] …)})      ;; merge the result back (default: result wins)
  • The sub-workflow's :green/exit (and :green/err/:green/trace) propagates like any step's — a failing sub-workflow halts or routes in the parent normally.
  • With the default identity :in, ambient keys (:green/event, :green/dry-run) flow in with opts, so lifecycle and dry-run work through arbitrary nesting. If a custom :in builds sub-opts from scratch, carry the ambient keys the sub-workflow needs; inherited advice is re-stamped automatically either way.
  • Fan out a composed step to run the same workflow many times with different scopes — e.g. one ZooKeeper-cluster workflow, fanned out per cluster with :in rewriting :zk/workdir/:zk/servers (see examples/multi-zookeeper).

Termination & slices

  • A branch terminates when there is no next step — wire-fn lists no successor (and no next-fn overrides), or next-fn returns empty.
  • Without next-fn, a step with :green/exit > 0 stops that path; in a fork, in-flight siblings finish their current step and the fork collapses without running the join.
  • The optional end step is a slice boundary: it runs, then the workflow stops regardless of wiring. Together with --start/--end on the CLI, any slice of the graph can be re-run — idempotency makes that safe.

5. Parallelism, fan-out & joins

Two sources of concurrency, one semantics:

  • Static fork: wire-fn lists several successors — each runs concurrently with the same opts.
  • Dynamic fan-out: next-fn returns several [step opts] pairs — e.g. one :zk/node per server in the desired state, each with its node assoc'd in.

Joins. When branches from different origins converge on the same step, the runner waits for all of them, then runs the join step once, with:

  • the opts captured at the fork point as the base, and
  • :green/branches — a vector of every branch's resulting opts.

Failure in a branch: running siblings finish, the join step is skipped, and the worst exit code (with its :green/err/:green/trace) propagates as the workflow result.

Limitation (v1). Convergence is detected by step name: branches arriving at the same qualified name join there. The scheduler holds a branch back from a join while any other live branch can still statically reach it (via wire-fn edges), so branches of different lengths join correctly. Dynamic next-fn detours invisible to the static graph may not be awaited.

Scheduler algorithm in plain English

The scheduler is basically a small fork/join workflow runner.
In simple words:

  1. Start with one live task
    • A “live” task means: “run step X with this opts map.”
    • Initially there is only the workflow’s :start step.
  2. Keep looping while there is work
    • The scheduler keeps two piles:
      • live: branches still running
      • finished: branches that reached the end
  3. Group live branches by step
    • If multiple branches are waiting at the same step, that may mean they need to join.
  4. Decide which steps are ready
    • Before running a step, the scheduler asks:
      • “Could another currently-running branch still reach this same step later?”
    • If yes, it waits, so the branches can join together.
    • This uses the static workflow graph from wire-fn.
  5. Run ready work in parallel
    • Ready steps are executed using future, so independent branches run concurrently.
  6. After a step runs, choose what happens next
    • No next step: branch is finished.
    • One next step: continue normally.
    • Multiple next steps: this is a fork; create one branch per successor.
  7. Handle joins
    • If branches from different origins arrive at the same step, the scheduler runs that step once.
    • It gives the join step:
      • the opts from the fork point
      • all branch results under :green/branches
  8. Handle failures inside forks
    • If one branch fails, siblings already running finish their current step.
    • Then the whole fork collapses.
    • The join is skipped.
    • The final result carries the worst exit code and all branch results.
  9. Finish
    • When there are no live branches left, the scheduler returns the final opts.
    • If multiple terminal branches exist, it returns the first failed one, otherwise the last successful one.

Short version: it repeatedly runs all safe-to-run steps in parallel, waits when branches may need to join, joins converged branches once, and collapses forks cleanly on failure.

6. Advice

green ports Emacs' advice-add/advice-remove to step functions and adds wf/advice-add-all/wf/advice-remove-all for all-steps advice. Advice is workflow-scoped and pure: adding or removing returns a new workflow value — no process-global registry, no teardown in tests.

(-> workflow
    (wf/advice-add :zk/node :before ::backend write-backend!)      ;; one step
    (wf/advice-add-all :around ::audit (fn [f opts] (f opts)))    ;; every step
    (wf/advice-remove :zk/node ::backend)
    (wf/advice-remove-all ::audit))
HowEffect (base f, advice a, input opts)Use cases
:around(a f opts) — full control; a decides if/how to call f.Dry-run skips, retries, timing/tracing, locks, or any wrapper that may call f zero, one, or many times.
:override(a opts) — replaces f entirely.Stub a step in tests, replace a default implementation, or swap behavior for one environment.
:before(a opts) for side effects (return ignored), then (f opts).Setup and prerequisites: write backend.tf, create directories, acquire a lock, validate inputs before side effects.
:after(f opts), then (a opts) for side effects (return ignored); f's return flows out.Audit, metrics, notifications, or cleanup that should observe the call but not change its result.
:before-while(and (a opts) (f opts)) — run inward only while advice returns truthy.Precondition gates: continue only while checks pass, such as maintenance-window, quota, or policy guards.
:before-until(or (a opts) (f opts)) — short-circuit with the advice result when it is truthy.Fast paths and no-ops: return a cached/already-converged/skip result before the expensive or destructive step runs.
:after-while(let [ret (f opts)] (if (green-true? ret) (a opts) ret)) — run advice only when the inward result is successful.Success-only follow-ups: run a verification, registration, or notification step only after f succeeded.
:after-until(let [ret (f opts)] (if (green-true? ret) ret (a opts))) — use advice as a fallback when the inward result failed.Recovery and fallback: handle a failed result by rolling back, repairing state, or supplying a fallback result.
:filter-args(f (a opts)) — transform the input map.Normalize or scope inputs: inject defaults, select a sub-map, add derived paths, or redact secrets before the step.
:filter-return(a (f opts)) — transform the result map.Normalize or enrich outputs: attach derived data, strip temp/private keys, redact values, or adapt a step's result shape.

For :after-while and :after-until, green-true? adapts Green's step contract to boolean advice semantics: a map with missing or zero :green/exit is true, while a positive :green/exit is false. Non-map values passed to green.advice/compose directly keep ordinary Clojure truthiness.

Stacking is Emacs-like: at equal :depth, the most recently added advice is outermost, and per-step/all-steps entries interleave in that single add order. A :depth property (-100..100, default 0) overrides add order: lower depths run farther outside, higher depths farther inside. Every advice carries an explicit :id; re-adding the same id replaces the old entry and removal is by id. The advice stack runs inside the runner's try/catch: a throwing advice fails the step through the normal :green/* contract.

Advice inheritance through composed workflows. When a workflow is embedded with wf/step, the enclosing run's effective advice is inherited by the nested run. Step names match flat at any depth: parent advice for :zk/node reaches an embedded workflow's :zk/node. At equal depth, inherited advice stacks outside the child's own advice; a parent entry with the same :id replaces the child entry, and explicit :depth can intentionally change the placement. Use wf/advice-plan with a workflow or workflow chain (for example [parent-wf cluster-wf]) to inspect the composed stack with provenance.

The library and examples dogfood advice heavily:

  • Backends — the OpenTofu backend is not hardwired: a :before advice writes the backend config into the working directory before the tofu step runs (the examples use the local-filesystem helper; S3/GCS helpers are alternatives, see §8). Swapping backends means swapping one advice.
  • Provider swaps — examples/once keeps :once/compute provider-agnostic; a :before advice writes either the digitalocean-mock or oci-mock HCL, both emitting the same {:ip :name :sudoer :user :uid} contract. Swapping clouds is the same re-add-under-one-id move as swapping backends.
  • Dry-run — green.dry-run/advise attaches an :around advice (id :green.dry-run/skip) to the side-effecting steps you name; when :green/dry-run is set (the CLI's --dry-run flag), those steps print dry-run: would run :zk/node (create) and are skipped with exit 0 — the rest of the graph still threads through.

7. The scaffolding DSL

Configuration files are generated from flat file specs: a seq of maps, one per file.

(require '[green.scaffold :as sc])

(sc/scaffold opts
  (for [node (:zk/servers opts)]
    {:template :zk/main.tf                          ;; resource path zk/main.tf
     :target   "{{workdir}}/nodes/{{node.id}}/main.tf"  ;; Selmer-rendered vs :data
     :data     {:workdir (:zk/workdir opts) :node node}}))
;; => opts with :green/exit 0 and :green.scaffold/written [targets…]
KeyMeaning
:templateQualified keyword resolved to a classpath resource: namespace dots become directories, the name is used verbatim — :zk/main.tf → zk/main.tf, :my.app/zoo.cfg → my/app/zoo.cfg. Templates live in a resource folder organized by qualified namespaces.
:targetOutput path; itself a Selmer template rendered against :data.
:dataThe map given to Selmer for both the template body and the target path.

scaffold is event-aware: on :green/event :delete it removes the targets (pruning each immediate parent directory if it became empty) instead of rendering them — the same specs that create the files name what to remove. Rendering is idempotent: same specs, same bytes.

Why flat specs (and not a nested tree): they're ordinary data built with ordinary seq functions; the Selmer-rendered targets are the manifest for delete; per-branch specs from a fan-out just concat.

8. OpenTofu integration

(require '[green.tofu :as tofu])

(tofu/tofu-step opts {:dir "work/nodes/1"})        ;; :output-key optional
  • Any non-:delete event (conventionally :create): tofu init, then tofu apply -auto-approve; afterwards tofu output -json is parsed and assoc'd into opts under :tofu/outputs by default. If you override :output-key, keep it namespaced by convention so observed outputs do not contaminate top-level desired state.
  • :delete: tofu init, then tofu destroy -auto-approve.
  • Non-zero tofu exits become :green/exit with stderr/stdout in :green/err.
  • Backends are explicit advices (see §6): green.tofu/backend-advice builds a :before advice writing backend.tf into the directory — backend type plus a flat attribute map (or a function of opts returning one). local-backend-advice is the helper used by the examples, and s3-backend-advice ({:bucket :key :region …})/gcs-backend-advice ({:bucket :prefix}) are shipped alternatives.

9. The CLI

./green <event> [-f|--file green.edn] [--start step] [--end step] [--dry-run]

./green create                        # green.edn, full graph
./green delete -f prod.edn            # different desired state
./green create --start zk/node --end zk/node   # re-run one slice
./green create --dry-run              # print what would run, touch nothing
  • The first positional argument is the lifecycle event, keywordized into :green/event.
  • --start/--end override the workflow's start/end for slice runs.
  • --dry-run stamps :green/dry-run true into opts; steps advised with green.dry-run (see §6) print what they would do and are skipped.
  • Missing event or state file → exit code 2 with a usage/error message in :green/err; exec prints it to stderr.
  • green.cli/run-cli returns the final opts (testable); green.cli/exec prints :green/err/:green/trace to stderr and calls System/exit with :green/exit.

The user's project ships a self-contained babashka script:

#!/usr/bin/env bb
(babashka.deps/add-deps
  '{:deps {io.github.amiorin/green {:git/sha "REPLACE_WITH_COMMIT_SHA"}}})
(babashka.classpath/add-classpath "resources")
(require '[green.cli :as cli] '[green.workflow :as wf])

(defn start-step [opts] opts)
(defn wire-fn [step]
  (case step
    :app/start [start-step]))

(def workflow (wf/workflow {:start :app/start :wire-fn wire-fn}))

(cli/exec workflow)                ;; forwards *command-line-args*, exits

11. Testing strategy

End-to-end tests run the real pipeline against resource-free OpenTofu configs when tofu is on PATH — HCL containing only locals and output blocks. tofu apply creates nothing real, needs no credentials, and finishes in milliseconds, yet exercises genuine render → apply → outputs → assert cycles, plus destroy for delete.

Flagship scenario: the fake ZooKeeper cluster

  1. green.edn declares the cluster: servers with id, name, IP, plus a work directory.
  2. :zk/start fans out via next-fn: one :zk/node branch per server, in parallel.
  3. Each branch scaffolds main.tf (locals: id/name/ip; outputs: the same) into its own directory and runs tofu apply; outputs merge back under :tofu/outputs. A :before advice writes the local-backend config first.
  4. The branches join at :zk/zoo-cfg, which reads every node's outputs from :green/branches and renders a zoo.cfg — listing all members — into each server's directory.
  5. delete runs tofu destroy for each node and removes the scaffolded files, symmetrically.

The suite runs under babashka (bb test) and the JVM (clojure -X:test), keeping the bb-compatibility promise honest.
